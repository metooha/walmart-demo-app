# RULE: Markdown File Organization

## Overview

All markdown documentation files MUST be organized in the appropriate directories. NO markdown files should exist in the project root except for essential project documentation.

## Directory Structure

### ✅ Allowed in Root

**ONLY these files are permitted in project root:**

- `README.md` - Project README (if exists)
- `CHANGELOG.md` - Version history (if exists)
- `LICENSE.md` - License file (if exists)

**All other .md files MUST be in organized folders.**

**Note**: AGENTS.md is now located in `guidelines/AGENTS.md`

### 📁 guidelines/

Primary location for all Living Design and component documentation.

#### guidelines/components/
**Purpose**: Component-specific usage guidelines

**File naming**: `ComponentName.md` (PascalCase matching component file)

**Examples**:
- `Button.md` - Button component guidelines
- `FilterChip.md` - Filter chip guidelines
- `IconButton.md` - Icon button guidelines

#### guidelines/design-system/
**Purpose**: Core design system documentation and token references

**Files include**:
- `DesignTokens.md` - Complete token reference
- `Color.md` - Color system
- `typography.md` - Typography system
- `Component-Inventory.md` - Available components
- `Guidelines.md` - General guidelines
- `Guidelines-react.md` - React guidelines
- Technical references (css-variables-source.md, etc.)

#### guidelines/implementations/
**Purpose**: Completed component implementation summaries

**File naming**: `[COMPONENT]-IMPLEMENTATION-SUMMARY.md` or `[FEATURE]-SUMMARY.md`

**Examples**:
- `ATTRIBUTION-DROPDOWN-SUMMARY.md` - Attribution filter implementation
- `COMPONENT-CONSOLIDATION-SUMMARY.md` - Button standardization summary
- `LD-TAG-IMPLEMENTATION-SUMMARY.md` - Tag component implementation

**When to add**:
- ✅ After completing a significant component implementation
- ✅ After a major refactor or consolidation
- ✅ When documenting "how we built X"
- ✅ For reference in future similar implementations

#### guidelines/migrations/
**Purpose**: Migration plans and implementation plans

**File naming**: `[COMPONENT]-MIGRATION-PLAN.md` or `[FEATURE]-PLAN.md`

**Examples**:
- `ICONBUTTON_MIGRATION_PLAN.md` - IconButton migration plan
- `THEME_SWITCHER_PLAN.md` - Theme switcher implementation plan
- `BUTTON-STANDARDIZATION-SUMMARY.md` - Button migration summary

**When to add**:
- ✅ Before starting a migration from old → new components
- ✅ Planning a major refactor
- ✅ Completed migration summaries

#### guidelines/rules/
**Purpose**: Enforced design system rules and standards

**File naming**: `RULE_[RuleName].md` (RULE_ prefix required)

**Examples**:
- `RULE_DesignTokenEnforcement.md`
- `RULE_IconUsage.md`
- `RULE_ThemeAddition.md`
- `RULE_MarkdownOrganization.md` (this file)

**When to add**:
- ✅ When establishing a mandatory practice
- ✅ When preventing common mistakes
- ✅ When documenting "how we MUST do X"

### 📁 .builder/

Builder.io and Fusion-specific files.

#### .builder/plans/
**Purpose**: Auto-generated implementation plans from Fusion

**Files**: Auto-generated with random names (e.g., `quiet-camp-v0znfygl.md`)

**DO NOT manually add files here** - these are generated by the system.

#### .builder/rules/
**Purpose**: Builder-specific rules

**Files**: `.mdc` format (markdown with code)

**Examples**:
- `deploy-app.mdc`
- `organize-ui.mdc`

### 📁 .fusion/rules/

Fusion AI assistant-specific rules.

**Examples**:
- `airtable-token-reference.md`
- `component-composition.md`
- `design-guidelines-mandate.md`

## File Organization Checklist

When creating or finding a markdown file, ask:

### 1. Is this a component guideline?
→ **Move to**: `guidelines/components/[ComponentName].md`

### 2. Is this design system documentation?
→ **Move to**: `guidelines/design-system/[filename].md`

### 3. Is this an implementation summary?
→ **Move to**: `guidelines/implementations/[FEATURE]-SUMMARY.md`

### 4. Is this a migration or implementation plan?
→ **Move to**: `guidelines/migrations/[FEATURE]-PLAN.md`

### 5. Is this an enforced rule or standard?
→ **Move to**: `guidelines/rules/RULE_[RuleName].md`

### 6. Is this project root documentation?
- ✅ **Keep in root** if: README.md, AGENTS.md, LICENSE.md, CHANGELOG.md
- ❌ **Move to guidelines** if anything else

## Enforcement

### ❌ VIOLATIONS (Auto-Fix Required)

```
❌ /PROJECT-ROOT/MY-COMPONENT-GUIDE.md
   → Move to: guidelines/components/MyComponent.md

❌ /PROJECT-ROOT/IMPLEMENTATION-NOTES.md
   → Move to: guidelines/implementations/Implementation-Notes.md

❌ /PROJECT-ROOT/DESIGN-TOKENS.md
   → Move to: guidelines/design-system/Design-Tokens.md
```

### ✅ CORRECT ORGANIZATION

```
✅ /AGENTS.md                                    (Essential project doc)
✅ /guidelines/components/Button.md              (Component guide)
✅ /guidelines/design-system/DesignTokens.md     (Design system doc)
✅ /guidelines/implementations/TAG-SUMMARY.md    (Implementation)
✅ /guidelines/migrations/BUTTON-MIGRATION.md    (Migration plan)
✅ /guidelines/rules/RULE_IconUsage.md           (Enforced rule)
```

## Common Mistakes

### ❌ DON'T: Create markdown in root

```bash
# BAD
touch /MY-NEW-FEATURE.md
touch /COMPONENT-GUIDE.md
```

### ✅ DO: Create in proper location

```bash
# GOOD
touch /guidelines/components/MyComponent.md
touch /guidelines/implementations/MY-FEATURE-SUMMARY.md
```

## When Unsure

If you're not sure where a markdown file belongs:

1. Check the content - what is it documenting?
2. Look at similar files - where are they located?
3. Default to `guidelines/design-system/` for general docs
4. Never leave it in project root (unless it's AGENTS.md/README.md)

## Update Checklist

After moving files:

- [ ] Update `guidelines/index.md` if structure changed
- [ ] Update any links in other files that referenced the old path
- [ ] Verify no broken links in documentation
- [ ] Run dev server to ensure nothing broke
- [ ] Search codebase for hardcoded file paths if file was referenced in code
